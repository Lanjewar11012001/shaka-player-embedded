1 # Media Pipeline Design
2 
3 This document describes how the core media pipeline works and how an app can
4 replace pieces of it to implement them differently.
5

7 ## Core types
8 
9 First is the frame types; these are used to store encoded
10 ([EncodedFrame][]) or decoded ([DecodedFrame][]) frames. Each object
11 represents a single frame and should maintain the lifetime of any pointers
12 returned; the data pointers must be static and are valid so long as the frame is
13 alive. To allow for shared lifetime management, all frames are stored in a
14 `std::shared_ptr<T>`. Apps should subclass these to store their own data and
15 ensure the frame data remains alive for the life of the object.
16 
17 The [EncodedFrame][] objects are created by the [Demuxer][]; the
18 [DecodedFrame][] objects are created by the [Decoder][] (if using
19 [DefaultMediaPlayer][]). The library-provided types don't care about the
20 specific implementation of the [EncodedFrame][] and [DecodedFrame][] types, so
21 apps are free to create subclasses and still use other library-provided
22 defaults.
23 
24 Each frame has a [StreamInfo][] that is used to tell what stream it comes from.
25 Frames coming from the same stream will have pointers to the same [StreamInfo][]
26 object. The [StreamInfo][] instance can be used to initialize decoders or
27 renderers.
28 
29 Buffers of frames are stored in the [Stream][] type. There is
30 [ElementaryStream][] and [DecodedStream][] for the [EncodedFrame][] and
31 [DecodedFrame][] types respectively. Alternatively apps can use
32 `Stream<T, OrderByDts>` to store a collection of their own types. A Stream
33 stores the frames in time order (PTS or DTS based on the `OrderByDts`
34 parameter). Streams only store one kind of frame (i.e. video-only or
35 audio-only).

38 ## MediaPlayer
39 
40 The [MediaPlayer][] is the top-level handler for all media interactions. It is
41 responsible for loading and playing content. The library provides the
42 [DefaultMediaPlayer][] which handles timeline tracking but provides other
43 plugins allowing apps to replace the [Decoder][] or [Renderer][]. The
44 [MediaPlayer][] operates in two modes, MSE and src= mode.
45 
46 In MSE mode, data is given to a [Demuxer][] and placed into an
47 [ElementaryStream][]. These streams are given to the [MediaPlayer][]. Then the
48 [MediaPlayer][] will handle reading, decoding, and rendering those frames.
49 
50 In src= mode, the [MediaPlayer][] is given a URL to read from. It is expected
51 to read that file to get the data. Then it will play it similar to MSE mode.
52 
53 When apps create a [Player](@ref shaka::Player) instance, they will need to pass
54 in the [MediaPlayer][] instance they want to play with. This will act similar
55 to the `<video>` element given in JavaScript.

58 ## Demuxer
59 
60 The [Demuxer][] is used when playing MSE content. This type is responsible for
61 reading the bytes from the stream and producing [EncodedFrame][] objects. This
62 type should do so synchronously and will be called on a background thread. This
63 is registered by creating an implementation of [DemuxerFactory][] and giving it
64 to [DemuxerFactory::SetFactory](@ref shaka::media::DemuxerFactory::SetFactory).
65 Passing `nullptr` to this method will cause the built-in [Demuxer][] to be used.
66 
67 One [Demuxer][] instance is created per input stream by the core library. For
68 example, one [Demuxer][] instance will handle video and the other will handle
69 audio. Changing the [DemuxerFactory][] will not change any existing [Demuxer][]
70 instances that are already being used.

73 ## Decoder
74 
75 The [Decoder][] is only used if using the [DefaultMediaPlayer][]. If providing
76 your own [MediaPlayer][] implementation, you don't need to use/provide a
77 [Decoder][]. The [Decoder][] is used to read [EncodedFrame][] objects and
78 decode them into [DecodedFrame][] objects. A [Decoder][] is only used to decode
79 one kind of media, but may be required to decode different input streams. It
80 should automatically check the [StreamInfo][] and re-initialize as required.
81 
82 Custom [Decoder][] implementations are given to
83 [DefaultMediaPlayer::SetDecoders](@ref shaka::media::DefaultMediaPlayer::SetDecoders).
84 Passing `nullptr` to either will cause the built-in decoders to be used.

87 ## Renderer
88 
89 The [Renderer][] is only used if using the [DefaultMediaPlayer][]. If providing
90 your own [MediaPlayer][] implementation, you don't need to use/provide a
91 [Renderer][]. They are also only used for MSE playback. The [Renderer][] is
92 used to read [DecodedFrame][] objects and either show them on the screen or play
93 on an audio device. This is expected to, on a background thread, pull frames
94 and update the data as needed. This can be done as part of audio callbacks, or
95 with an explicit background thread. The [Renderer][] will be given the
96 [MediaPlayer][] instance so it can read the current time, playback rate, etc
97 that are needed to show the content.
98 
99 On Desktop, we provide an
100 [SdlAudioRenderer][], [SdlManualVideoRenderer][], and [SdlThreadVideoRenderer][]
101 types for use with SDL. On iOS, we don't expose the iOS renderers; we expect
102 iOS users to use the [ShakaPlayerView](@ref ShakaPlayerView).
103 
104 [Renderer][] instances are given to the [DefaultMediaPlayer][] constructor.

107 ## Encryption support
108 
109 Encryption support comes through the [Eme Implementation][] and
110 [ImplementationFactory](@ref shaka::eme::ImplementationFactory) types. Once an
111 EME instance is created, it is given to the [MediaPlayer][]. If using the
112 [DefaultMediaPlayer][], it will be given to the [Decoder][] and the [Decoder][]
113 will decrypt and decode the frame as needed. In this case, it is assumed the
114 frames are decoded into the clear. This allows supporting secure
115 decrypt-decode, but doesn't allow for secure rendering.
116 
117 Apps can use secure rendering by providing their own [MediaPlayer][] instance.
118 Since this case doesn't allow clear frames in memory, the [DefaultMediaPlayer][]
119 cannot handle it. You'll need to handle all the media playback.

122 ## Removing defaults
123 
124 When running `./configure`, you can disable default media components you don't
125 need when providing your own replacements. When these are used, the app will
126 need to provide their own overrides or playback won't work.
127 
128 - `--no-media-player`: Removes the [DefaultMediaPlayer][]. This also implies
129  `--no-decoder`, `--no-sdl-audio`, and `--no-sdl-video`.
130 - `--no-demuxer`: Removes the default [Demuxer][] implementation.
131 - `--no-decoder`: Removes the default [Decoder][] implementation.
132 - `--no-sdl-audio`: Removes the [SdlAudioRenderer][].
133 - `--no-sdl-video`: Removes the [SdlManualVideoRenderer][] and
134  [SdlThreadVideoRenderer][]. This is the default on iOS.
