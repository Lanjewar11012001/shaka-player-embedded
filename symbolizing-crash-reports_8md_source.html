1 Symbolizing Crash Reports
2 =========================
3 
4 This doc describes how to manually symbolize a crash report on a release binary.
5 This should be read both by partners who send crash reports and for us to
6 actually symbolize it. This process can only be done if we have the debug info
7 for the binary. This only happens for official builds that we release. If you
8 build from source, you can do this yourself with the debug info in the build
9 directory.
10 
11 Release binaries are optimized and don't have debug info. This means there is
12 no way to get a useful stack trace or to get useful info from local variables.
13 Going from a memory address to a symbol and line number is symbolizing. This
14 allows us to see the real stack trace.
15 
16 
17 What is needed
18 --------------
19 
20 First, we obviously need the stack trace. The best option is a whole crash
21 dump (e.g. a "core" file on linux). If you provide that, you're done. But we
22 can work with a copy of the stack trace text and some extra info.
23 
24 It is important that the entries show the memory address by itself or something
25 like `foo.so+0x1234`. We need the exact bytes. Note that
26 `ExportedMethod+0x123` is not useful, it needs to be absolute or relative to the
27 dynamic library.
28 
29 Next we'll need which exact library you used; this includes the version string,
30 OS, and architecture. We'll also need the build ID of the library. Run one of
31 the following commands to get it:
32 
33 ```sh
34 # On Mac (or iOS with dynamic library):
35 dwarfdump -u <library.so>
36 # On iOS:
37 dwarfdump -u <library.framework/library>
38 # On Linux:
39 readelf -n <library.so> | grep "Build ID"
40 ```
41 
42 Windows doesn't have an exact build ID. Instead just give the file version.
43 This can be retrieved by right-clicking on the file and selecting "Details".
44 Or in a PowerShell window, run:
45 
46 ```sh
47 (Get-Command ./library.dll).FileVersionInfo.FileVersion
48 ```
49 
50 ### Load address
51 
52 Lastly, we'll need the load address of the library in your executable. This is
53 the address in the binary's memory where our library is loaded. Since this is
54 a shared library, it can appear anywhere in memory. We need to map the address
55 in the stack trace to relative to our library so we know what the code is.
56 
57 It is IMPORTANT to get the load address for the same invocation the stack trace
58 is from. Dynamic libraries can appear at different locations, so you need to
59 use the same run to get the load address and the stack trace.
60 
61 If you have a debugger open in Xcode, select
62 `Debug > Debug Workflow > Shared Libraries...`. Then give us the address for our
63 shared library.
64 
65 If you have a debugger open in Visual Studio, open the "Modules" tab (if not
66 open, select `Debug > Windows > Modules`). Then give us the "Address" for our
67 library.
68 
69 If you have a crash log from an iOS device, there is a section for
70 `Binary Images:` that lists the shared libraries. We need the one that matches
71 the build ID you got before. To get logs for an attached device, open Xcode
72 and select `Window > Devices and Simulators`, select your device and click the
73 `View Device Logs` button.
74 
75 If you have a crash dump for Mac or iOS and are using `lldb`, enter `image list`
76 and give us the address for our library. It will probably be near the top.
77 
78 If you have a core dump for Linux, open it in `gdb` and enter `info share`
79 and give us the address for our library. It will probably be near the top.
80 
81 If you have a core/minidump for Windows, open it in `cdb` or `windbg` and enter
82 `lm` and give us the start address for our library.

85 Symbolizing the stack trace
86 ---------------------------
87 
88 First you need the debug info for the build. For Mac and iOS this is a `.dSYM`
89 file. On Linux, we just strip the shared library, so it is just a `.so` file
90 that has been renamed to `.so.debug`. Run the commands above to get the build
91 ID and verify it matches what the report gave us.
92 
93 For Mac and iOS, run the following command. This will print the function name,
94 file, and line number for each address given. If the address can't be found it
95 will just print the address itself.
96 
97 ```sh
98 atos -o <library.dSYM> -l <LOAD_ADDRESS> <ADDR1> <ADDR2> ...
99 ```
100 
101 For Linux, subtract the load address from the address and enter it (as a hex
102 number) into the following command. If it doesn't work, try subtracting one
103 from the number.
104 
105 ```sh
106 addr2line -Cf -j .text -e <library.so.debug> <OFFSET1> <OFFSET2> ...
107 ```
108 
109 For Windows, you'll need the [Windows debugger tools][1] (not Visual Studio).
110 This usually comes with the Windows SDK or the Windows Driver Kit. You then
111 need to run `dbh.exe`, an interactive symbol viewer/searcher:
112 
113 [1]: https://docs.microsoft.com/en-us/windows-hardware/drivers/debugger/
114 
115 ```
116 $ dbh.exe -v library.pdb
117 
118 > base <LOAD_ADDRESS>
119 > addr <ADDR1>
120  name : my_symbol
121  addr : <ADDR1>
122  size : 0
123  flags : 400000
124  type : 0
125 modbase : 7ff123456789
126  value : 0
127  reg : 0
128  scope : SymTagNull (0)
129  tag : SymTagPublicSymbol (a)
130  index : 12ab
131 
132 > laddr <ADDR1>
133  file : C:\users\build\src\foo.cpp
134  line : 123
135  addr : <ADDR1>
136  key : 000000000000000
137 ```
138 
139 Note that `laddr` might not work for third-party static libraries if the line
140 info isn't available; but the `addr` command should always work.

143 Symbolizing crash dumps
144 -----------------------
145 
146 If you have a crash dump from an unknown program, you can still get debug info
147 from it. This also can be done to add symbols to a debugger run if there was
148 problems loading the symbols.
149 
150 ### Linux
151 
152 First, load the core dump (`gdb -c core`). Then find our library in the loaded
153 shared library list (`info share`) and remember the start address. Last, load
154 the symbols for our library (`add-symbol-file library.so.debug 0x12345`). You
155 should now have symbols for all of our parts and can move around our code like
156 normal.
157 
158 ### Mac
159 
160 For OSX core dumps, load the core dump (`lldb -c core`). Then add our dSYM
161 file (`add-dsym library.dylib.dSYM`).
162 
163 ### Windows
164 
165 For Windows core/mini dumps, you can add the path to the folder containing the
166 .pdb on the command-line. For example: `cdb -z core.dmp -y symbol_folder` or
167 `windbg -z core.dmp -y symbol_folder`. If you have an existing session and
168 don't have symbols, the run `.sympath+ symbol_folder` and `.reload`. Be sure
169 to give the folder name and that the .pdb file hasn't been renamed.
