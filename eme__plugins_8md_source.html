1 # Custom EME Plugins
2 
3 This document describes how to create and register custom EME plugins. Defining
4 a custom EME implementation allows the app to handle decrypting the media
5 content without handling media-specific features. As of now, the EME
6 implementations cannot handle decoding; they can only decrypt the bytes. This
7 means you can't implement secure decrypt-decode or secure rendering.
8 
9 
10 ## Overview
11 
12 First, you need to create the custom implementation types. You need to define
13 derived types for [Implementation](@ref shaka::eme::Implementation) and
14 [ImplementationFactory](@ref shaka::eme::ImplementationFactory). There will be
15 a single, global instance of the factory that is used to create instance of the
16 Implementation. Each playback instance will get its own Implementation instance
17 and there can be multiple alive at once if there are multiple Player instances.
18 
19 ### Registering your plugin
20 
21 Once you have written the type, you need to compile it into some library. You
22 have two options:
23 
24 First, you can compile your code into your app. You can do so any way you like.
25 Then, before you create a Player instance, you need to create a global instance
26 of your ImplementationFactory and register it with the
27 [ImplementationRegistry](@ref shaka::eme::ImplementationRegistry). This factory
28 instance must live for the duration of the app.
29 
30 Second, you can compile your code into a separate library. Then you use the
31 plugin definition below to register with our build system. When you build Shaka
32 Player Embedded, our build system will include your library and include the
33 plugin into the Embedded library. The implementation(s) will be automatically
34 registered during library startup.
35

37 ## Plugin Definition
38 
39 If you want, you can create a JSON definition of the plugin and have it built
40 into the library. This means the plugin will be auto-registered and you just
41 have to use the library like normal. This also allows you to keep the library
42 and the plugin together and avoid versioning problems.
43 
44 To define a plugin, you write a JSON file that describes your plugin. It should
45 contain one or more libraries that will be built into the library. Then you
46 indicate the headers where the implementations are found and we will generate
47 code to register those types.
48 
49 To register your JSON file, pass the `--eme-impl` flag to `./configure`. You
50 can pass this multiple times for multiple implementations:
51 
52 ```sh
53 ./configure --eme-impl my-eme-def.json
54 ```

118 ### Dynamic libraries
119 
120 It is suggested to use a static library to make building easier. But you can
121 use a dynamic library or a framework instead. For iOS builds, the library
122 or framework will be copied into the compiled framework; for non-iOS builds,
123 you are responsible for copying the dynamic library to the install directory
124 so the library loader can find it at runtime.
125 
126 #### Factory requirements
127 
128 Each implementation factory defined by the plugin must publicly implement the
129 ImplementationFactory type and must have an empty constructor. The factory will
130 be allocated using `new`. For static libraries, the type doesn't need to be
131 exported.
132 
133 #### Paths
134 
135 File paths in the JSON file can be absolute; if they are relative, they are
136 relative to the JSON file. File paths can include format specifiers to give
137 different paths based on what is currently being built. The format specifier is
138 in the format `%(name)s` where name is one of the below names. For example,
139 `foo/%(config)s/bar` may give `foo/Debug/bar`.
140 
141 | Name | Description | Possible values |
142 | ---- | ----------- | --------------- |
143 | config | The current configuration name | `Debug` or `Release` |
144 | arch | The target CPU architecture | `x86`, `x64`, `arm`, or `arm64` |
145 | os | The target OS | `linux`, `mac`, `win`, `ios`, or `android` |
146 
147 #### Building plugin from source
148 
149 It is possible to build your plugin from source instead of from a static
150 library. If you pass the `sources` field, those files will be compiled into
151 the library. This allows access to the library internals. For example, this
152 can be used so your plugin can use the
153 [FileSystem](@ref shaka::util::FileSystem) type to avoid duplicating code. Note
154 that internals are not ABI or API stable, so they may change at any time.
